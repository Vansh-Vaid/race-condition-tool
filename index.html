📚 How It Works

⚠️

What is a Race Condition?

A race condition occurs when two or more threads access a shared variable concurrently and at least one of them writes to it. The final result depends on the unpredictable order of execution — meaning the program may produce different outputs each time it runs.

Example: Thread A reads counter = 5, Thread B reads counter = 5, both compute 5 + 1 = 6 and write 6. The counter should be 7, but ends up as 6 — a lost update.
📐

Bernstein's Conditions

Two operations in different threads form a potential race if all of these are true:

  1. They belong to different threads
  2. They access the same shared variable
  3. At least one is a WRITE operation
  4. They can execute concurrently (no ordering constraint)

If only READs are involved, there is no race — reading shared data is safe.

🔒

Lockset / Eraser Algorithm

The Eraser algorithm (from the 1997 Savage et al. paper) tracks which locks are held when each operation executes. For each pair of conflicting operations:

  • Compute the lock set of each operation (all locks held at that point)
  • If the lock sets have a non-empty intersection → protected (no race)
  • If the lock sets have no common lock → unprotected → RACE!

This is a conservative (may report false positives) but effective technique used in real-world race detectors.

🏷️

Race Severity Levels

CRITICAL Write-Write (WW) Both threads write — guaranteed data corruption
WARNING Read-Write (RW/WR) One reads while other writes — may see stale/partial data
SAFE Protected by mutex Common lock ensures mutual exclusion — no race
🛡️

How to Fix Race Conditions

The most common solution is to use mutual exclusion (mutex):

  1. LOCK(mutex) before accessing the shared variable
  2. Perform your READ / WRITE operations
  3. UNLOCK(mutex) when done

When all threads acquire the same lock before accessing a variable, only one thread can access it at a time — eliminating the race.

Try it: Load the "Bank Transfer" preset, notice the Sender uses a lock but the Receiver doesn't. The tool will detect a race. Add LOCK/UNLOCK around the Receiver's operations to fix it!
📝

Operation Types Guide

READ Read from a shared variable (e.g., temp = counter)
WRITE Write to a shared variable (e.g., counter = temp + 1)
LOCK Acquire a mutex lock (e.g., pthread_mutex_lock(&m))
UNLOCK Release a mutex lock (e.g., pthread_mutex_unlock(&m))
